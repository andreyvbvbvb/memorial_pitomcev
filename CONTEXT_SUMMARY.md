## 1) Архитектура и инфраструктура
- **Монорепо pnpm workspaces**:
  - `apps/web` — Next.js App Router.
  - `apps/api` — NestJS + Prisma + PostgreSQL.
  - `packages` — общие модули, используется минимально.
- **Деплой**: Timeweb Apps через Docker Compose, сервисы `web` и `api`.
- **Основной публичный домен**: `https://мяугав.com`.
  - ASCII/punycode форма для server env и Docker/Timeweb: `https://xn--80aeb9a9a9d.com`.
  - При общем домене web+api фронт должен ходить на API через `NEXT_PUBLIC_API_URL=/api`, а Next proxy прокидывает `/api/*` во внутренний `INTERNAL_API_URL=http://api:3001`.
- **Локально**: web `3002`, api `3001`.
- **Хранилище**: Timeweb Object Storage / S3-compatible.
- **3D**: React Three Fiber + drei + three, Draco decoder лежит локально в `apps/web/public/draco`, без зависимости от внешнего CDN.
- **Canvas**: везде выставлен `dpr={1}` для снижения нагрузки на GPU.
- Актуальные версии web после обновления безопасности:
  - Next.js `15.5.16`;
  - React `19.2.3`;
  - React DOM `19.2.3`;
  - `@react-three/fiber` `^9.6.1`;
  - `@react-three/drei` `^10.7.7`.

## 2) Основные сущности и фичи
- Пользователи, питомцы, мемориалы, фотографии, подарки, размещения подарков, wallet.
- Мемориал может быть публичным или приватным.
- Публичные мемориалы отображаются на карте.
- Есть внутренняя валюта, пополнение баланса и покупка подарков.
- При регистрации пользователь получает стартовый бонус 150 монет.
- Все ключевые операции баланса пишутся в `WalletTransaction`: бонус регистрации, пополнение, создание/продление мемориала, покупка подарка, админское начисление и начальный баланс при bulk-create.
- В профиле показывается история пополнений и трат с датой, суммой, остатком после операции и описанием.
- 20% от пополнения баланса начисляется в благотворительный фонд проекта.
- Цены аренды мемориала хранятся в БД (`MemorialPlanPrice`) и могут меняться через админку.
- Цены подарков хранятся в `GiftCatalog.price` и могут меняться через админку без перезаписи при автосинхронизации ассетов.
- Есть страница благотворительности с отчетами и админским созданием отчетов с фото через S3.
- Есть страница `/news` для новостей и предупреждений о технических работах. Owner/admin могут публиковать новости прямо на этой странице и прикладывать фотографии, которые хранятся в S3.
- Для документов есть история PDF-редакций через `DocumentRevision`.
- Есть загрязнение мемориала: грязь накапливается по времени с момента создания/последней очистки, хранение состояния на сервере.
- По умолчанию пользователь может создать максимум 5 мемориалов; для owner действует лимит 10000. Лимит можно менять в `/admin/sql` по email.
- Мемориалы имеют срок активности (`Memorial.activeUntil`) и могут быть деактивированы без физического удаления данных.
- Удаление мемориала пользователем — soft-delete: мемориал скрывается из списков/карты/лимитов, но записи остаются в БД.

## 3) Доступы и роли
- Есть уровни доступа:
  - `OWNER` — владелец проекта: `andreyvbvbvb@gmail.com` / login `andreyvbvbvb`.
  - `ADMIN` — админ через поле `role=ADMIN` в БД.
  - `USER` — обычный пользователь.
- `OWNER` может назначать и снимать админов через `/admin/sql` по email.
- `ADMIN` имеет доступ к админским инструментам, но не может:
  - понижать owner;
  - назначать новых admin;
  - менять owner через SQL/смену пароля.
- Временные калибровочные инструменты в `/create` показываются только `OWNER` и `ADMIN`.

## 4) Auth, документы и профиль
- При регистрации пользователь должен принять пользовательское соглашение и публичную оферту.
- Для уже существующих пользователей после авторизации показывается обязательное окно принятия документов, если флаги не проставлены.
- Приватные web-страницы защищены middleware по cookie `access_token`:
  - `/my-pets`;
  - `/profile`;
  - `/admin/sql`.
- `/create` сейчас публичный, чтобы пользователь мог начать создание без входа. Все действия, которые сохраняют данные на сервере, требуют авторизации на уровне API.
- Если неавторизованный пользователь открывает приватный URL напрямую, middleware редиректит на `/auth?next=...`; после входа `/auth` возвращает пользователя на `next`, если путь локальный.
- API дополнительно защищает owner-sensitive endpoints через Nest guard по cookie/JWT:
  - создание/редактирование/удаление питомца и мемориала;
  - чтение/создание/удаление черновиков;
  - фото и map-preview;
  - очистка/продление мемориала;
  - дарение подарков;
  - wallet;
  - профиль пользователя;
  - owner-списки `/pets?ownerId=...`.
- Публичный доступ к чужим мемориалам идет только для публичных записей. Для списка публичных мемориалов владельца используется `/pets?ownerId=...&visibility=public`.
- `/about` — страница о проекте со ссылками на документы.
- Пользовательское соглашение вынесено на `/terms`.
- Публичная оферта вынесена на `/offer`.
- `/terms` и `/offer` теперь состоят из вкладок:
  - актуальный текст;
  - история PDF-редакций с датой загрузки и ссылкой на скачивание.
- В `/admin/sql` owner/admin может загрузить новый PDF-документ, выбрав тип `terms` или `offer`.
- На страницах `/terms` и `/offer` кнопки возврата на `/about` убраны; переход к документам идет ссылками из `/about` и auth-окон.
- Все ссылки принятия документов в auth-окнах ведут на `/terms` и `/offer`.
- На `/auth` нет кнопки входа в header.
- На `/profile` убраны кнопки `Мои питомцы` и `Создать мемориал`.
- На `/profile` карточка настроек не показывается до загрузки данных профиля, чтобы не было промежуточного дергания старого/пустого состояния.
- На `/profile` используется одно окно с вкладками:
  - `Профиль` — данные аккаунта и редактирование;
  - `Баланс` — текущий баланс и история операций;
  - `Подарки` — галерея подарков.
- В профиле есть история баланса и галерея подарков:
  - подарки, которые пользователь дарил;
  - подарки, которые получили мемориалы пользователя.
- Восстановление пароля: показывается инструкция написать на почту поддержки `meowgav.service@mail.ru`; этот контакт также указан на `/about`.

## 6) Создание мемориала `/create`
- Мастер сейчас состоит из **2 шагов**.
- **Шаг 1**:
  - один большой общий декоративный блок;
  - внутри слева отдельное большое окно выбора "души" питомца с 3D-превью на фоне неба;
  - внутри справа декоративный блок основных данных;
  - поля: имя, вид питомца, дата рождения, дата ухода;
  - выбирается один цвет души: цвет ядра и цвет ауры синхронизированы;
  - цвета души меняются сразу при hover/focus по цвету и фиксируются при клике;
  - есть 16 базовых цветов и кастомный выбор через color picker;
  - кнопка `Продолжить` со стрелкой-анимацией;
  - фон должен быть единым с header; на узких экранах шаг может скроллиться внутри своего viewport, чтобы блоки не ломались.
- До готовности 3D-превью души на шаге 1 показывается fullscreen blur/loading, чтобы пользователь видел загрузку и форма не выглядела зависшей.
- Между шагами — непрозрачный fullscreen loading screen:
  - progress bar;
  - подсказки сервиса;
  - подсказки управляются через админку (`LoadingTip`).
- При открытии редактирования мемориала через `/create?edit=...` используется тот же fullscreen preload, чтобы 3D и ассеты не появлялись без начальной загрузки.
- При переходе из редактирования по кнопке header `+ создать` страница `/create` перемонтируется с чистым состоянием и открывает шаг 1.
- **Шаг 2**:
  - 3D-превью на весь экран как фон;
  - справа — редактор деталей мемориала;
  - слева снизу — кнопки окон: цвет души, основные данные, история/эпитафия, карта/маркер, фото;
  - кнопка цвета души стоит первой сверху, пока пользователь ее не открыл — подсвечивается с `!`;
  - окно "Цвет души" позволяет менять единый цвет души; popover кастомного цвета в этом окне открывается вверх, чтобы не уходить под нижнюю границу экрана;
  - окна от левых кнопок располагаются справа от панели кнопок и не должны накладываться на сами кнопки;
  - справа снизу — кнопка `Завершить`, открывает модал проверки и оплаты;
  - баланс подгружается заново при каждом открытии проверки.
- Мобильная версия шага 2 и редактирования:
  - экран делится на верхнее 3D-превью и нижнюю рабочую область редактора;
  - нижняя область занимает всю ширину экрана без отдельной декоративной рамки;
  - слева — вертикальная панель кнопок разделов, справа — контент активного раздела;
  - кнопка `Сохранить/Продолжить` вверху открывает модальное окно выбора `Сохранить черновик` или `Завершить`;
  - кнопка-книжка вверху открывает локальное popover-меню, а не `/menu`;
  - в блоке карты сама карта остается внутри нижней области, а широта/долгота/публичность показываются поверх 3D-превью только при активном блоке карты;
  - списки домиков/текстур, поверхностей/сезонов и маркеров имеют собственные scroll-зоны, чтобы не скроллилась вся страница;
  - в мобильном выборе домика текстуры показываются компактными swatch-квадратами/вертикальным списком, а не большими фото-карточками;
  - в мобильном блоке маркеров количество колонок адаптируется к ширине экрана;
  - фокус/движение камеры сохраняется только для выбора домика и поверхности; остальные категории деталей не должны двигать камеру.
- Окна шага 2 закрываются кликом вне окна.
- Кнопки слева подсвечиваются, пока пользователь хотя бы раз не открывал соответствующий блок; вместе с подсветкой показывается `!`.
- Временные калибровочные слайдеры позиции/масштаба домика видны только admin/owner.
- Для публикации требуется минимум одна фотография питомца.
- Если пользователь начал создание без входа, кнопка публикации в проверочном окне сначала открывает авторизацию/регистрацию; после успешного входа публикация продолжается от имени вошедшего пользователя.
- Если после авторизации монет достаточно, публикация может продолжиться автоматически; если монет не хватает, показывается окно пополнения, после закрытия можно сохранить мемориал как черновик.

## 7) Данные в `/create`
- **Основные данные** можно открыть и поправить на шаге 2.
- **История/эпитафия** — отдельное окно.
- **Маркер**:
  - карта слева;
  - справа категории маркеров по видам питомцев;
  - категории — вертикальные иконки с tooltip;
  - можно кликнуть по карте или вручную ввести координаты;
  - есть чекбокс публичности с tooltip.
- **Фото**:
  - максимум 10 фото;
  - до 6 МБ каждое;
  - пользователь выбирает фото-обложку для popup маркера;
  - служебный 3D-скрин мемориала не попадает в галерею и используется отдельно для списка на `/map`.
  - скрин делается от фиксированной базовой позиции камеры, а не от текущего пользовательского ракурса.
- **Поверхность / сезоны**:
  - seasons доступны как варианты terrain;
  - переключатель авто-смены сезонов по умолчанию выключен;
  - сезоны представлены цветными квадратиками.
- **Превью подарков**:
  - чекбокс `Посмотреть с подарками` показывает пример заполнения слотами подарков.
  - набор случайных моделей подарков для preview создается один раз на текущее окно создания/редактирования и сохраняется при включении/выключении показа, чтобы подарки не менялись хаотично.

## 7.1) Черновики мемориалов
- Черновики хранятся на сервере в модели Prisma `MemorialDraft`, а не в браузерном IndexedDB.
- API черновиков:
  - `GET /pets/drafts`;
  - `GET /pets/drafts/:id`;
  - `POST /pets/drafts`;
  - `DELETE /pets/drafts/:id`.
- Все draft endpoints защищены `AuthGuard`; пользователь может читать и менять только свои черновики.
- Создание без входа возможно только как локальный текущий процесс в форме `/create`. Сохранение черновика всегда требует входа или регистрации.
- Если неавторизованный пользователь нажимает `Сохранить черновик`, открывается auth-модал; после успешной авторизации черновик сохраняется на сервер и пользователь попадает в `/my-pets`.
- В черновиках не хранятся фотографии. UI предупреждает, что фото сохраняются только при публикации мемориала.
- Открытие черновика идет через `/create?draft=...`; черновик загружается без всплывающего уведомления `Черновик загружен`.
- После успешной публикации мемориала связанный черновик удаляется.
- Черновики показываются только в карточном режиме `/my-pets`.
- На `/my-pets` есть переключатель `Показывать черновики`; в 3D-режиме `/my-pets` и на `/map` черновики не отображаются.

## 8) 3D: terrain, house, parts
- Terrain и house — отдельные GLB.
- В terrain обязателен `dom_slot`.
- Домик ставится в `dom_slot`, затем к нему применяются:
  - fit-scale;
  - индивидуальная позиция/поворот/scale из `apps/web/lib/house-layout.ts`;
  - при необходимости override по terrain.
- В 3D-превью весь мемориал слегка "плавает" через отдельную float-группу: текущая амплитуда около `0.065` по X/Z и `0.033` по Y, чтобы движение было заметным, но не мешало выбору слотов.
- Текущие группы домиков: `budka_*`, `kotik_*`, `mat_*`.
- Названия в UI:
  - `budka_N` → `Будка N`;
  - `kotik_N` → `Котик N`;
  - `mat_N` → `Матрас N`.
- Поддерживаются house textures через отдельные GLB вида `DOM_{baseId}__{texture}.glb`.
- Для базовой текстуры используется `DOM_{baseId}__base.glb` и она тоже показывается в выборе текстуры.
- Иконки домиков и текстур — заглушки/картинки в `apps/web/public/memorial/options/house` и `house-texture`.
- Для всех домиков `mat_*` используется единый канонический размер fit-scale от `DOM_mat_1__base.glb` (`0.848703 x 0.29245 x 0.979797`), чтобы разные текстуры/варианты `mat_1__...` не масштабировались по собственному bbox и слоты мисок оставались на одинаковых итоговых позициях.
- Канонический fit-size задается в `apps/web/lib/house-layout.ts` через `getHouseScaleFitSizeOverride` и применяется в `/create`, `/my-pets` 3D и `/map` 3D.

## 9) Слоты деталей домика
Слоты берутся автоматически из GLB домика по именам узлов:
- `roof_slot`
- `wall_slot`
- `sign_slot`
- `frame_left_slot`
- `frame_right_slot`
- `mat_slot`
- `bowl_food_slot`
- `bowl_water_slot`

Если слот отсутствует в GLB, соответствующая категория выбора в редакторе не показывается.
- Для `budka_1` действует отдельный fit-limit деталей:
  - `bowl_food_slot` и `bowl_water_slot` — не больше `0.18 x 0.18` по X/Z;
  - `mat_slot` — не больше `0.33 x 0.33` по X/Z.
- Эти ограничения задаются в `apps/web/lib/house-layout.ts` через `getHousePartFitBounds` и применяются одинаково в `/create`, `/map` 3D и `/my-pets` 3D.

## 10) Подарки
- Слоты подарков в моделях:
  - `gift_{type}_{index}` — основной формат;
  - legacy `gift_slot_1..` читается как `ground`.
- Дефолтный список слотов — 10, но рендерятся только реальные anchors в GLB.
- Заглушка слота: `apps/web/public/models/gifts/slot_placeholder.glb`.
- Типы подарков включают `toy`, `meal`, `birds`, `flower`, `candle`, `star` и др. по структуре папок.
- Иконки подарков: `apps/web/public/gifts_icons/{code}.png`.
- В списке подарков карточка: картинка занимает карточку, цена в кружке снизу по центру.
- Срок подарка выбирается ползунком.
- Истекшие подарки не удаляются из БД, а деактивируются через `GiftPlacement.isActive=false`, `deactivatedAt`, `deactivationReason=expired`.
- На странице мемориала есть чекбокс подсветки доступных мест под подарки; при выборе конкретного подарка подсвечиваются подходящие слоты независимо от чекбокса.
- В мобильной версии страницы мемориала подсветка мест подарков включена по умолчанию и отдельная плашка `Подсвечивать места для подарков` скрыта.
- В мобильном блоке подарков карточки подарков имеют квадратное изображение, цена вынесена в круг по центру нижнего ребра, а нижняя зона срока и кнопки `Подарить` уплотнена, чтобы не обрезаться границами блока.
- Автогенерация подарков обновляет web/api generated-файлы при `prebuild`.

## 11) Загрязнение мемориала
- На странице мемориала грязь считается сервером от даты создания или последней очистки.
- Каждые 2 недели добавляется следующий уровень, максимум 4.
- Загрязнение теперь хранится по слотам: можно очистить отдельное пятно, остальные остаются.
- Клик по конкретной модели грязи удаляет только это пятно и показывает toast с количеством оставшихся пятен.
- Если очищено последнее пятно, счетчик загрязнения сбрасывается как при полной очистке; если пятна еще остаются, расписание появления следующего пятна не сбрасывается.
- Логика добавления нового пятна идет по порядку слотов и пропускает уже занятые, чтобы частичная очистка не ломала очередность.
- Слоты грязи в GLB называются `dirt_slot_1..4`: слоты 1-2 планируются на поверхности, 3-4 на домике.
- Поддерживается несколько моделей грязи:
  - `dirt_n.glb` — может попасть в любой слот;
  - `dirt_n_m.glb` — может попасть только в слот `m`.
- Грязь отображается на странице мемориала, в 3D-режиме `/my-pets` и в 3D-режиме `/map`.
- Служебный preview для карты переснимается при появлении/очистке грязи, чтобы карточки карты отражали актуальное состояние.
- Визуальные ограничения грязи в `DirtSlotAttachments`:
  - для слотов поверхности (`dirt_slot_1`, `dirt_slot_2`) максимальный размер по X/Z — `1.4`;
  - для слотов домика (`dirt_slot_3`, `dirt_slot_4`) максимальный размер — `0.5`;
  - материал грязи принудительно перекрашивается в светло-коричневый `#a88766`, текстурные карты сбрасываются.

## 12) Душа питомца
- Добавлена текущая версия "души" питомца.
- Реализация не использует отдельный GLB: модель процедурная в `apps/web/components/PetSoul.tsx`.
- Визуально душа сейчас состоит только из двух слоев:
  - основное цветное ядро: sphere radius `0.199`, opacity `0.96`, normal blending;
  - средняя прозрачная волнистая аура: sphere radius `0.34`, opacity `0.18`, additive blending.
- Убраны внешняя прозрачная сфера, световые кольца, кометный след, лучевые вспышки, туманные ленты, орбитальные частицы, внутренние искры и маленький вертикальный внутренний сгусток.
- Аура деформируется в `useFrame` через изменение вершин `sphereGeometry`: контур слегка "дышит" и меняет форму.
- Пульсация:
  - ядро: `1 + sin(t * 2.4) * 0.08`;
  - аура: `1 + sin(t * 1.7 + 0.8) * 0.065`.
- Выбор второго цвета убран. В UI выбирается один цвет души; `glowColor` сохраняется равным `color` только для совместимости со старым `sceneJson`.
- Есть 16 базовых цветов в `SOUL_COLOR_OPTIONS` и кастомный выбор через color picker.
- Базовая позиция души считается относительно поверхности, а не домика.
- Движение души ограничено около домика и поверхности: idle/arrival не проходят через центр домика и не опускаются ниже безопасного уровня поверхности; farewell строит облет вокруг мемориала и затем уводит душу в домик.
- Idle-движение души теперь потоковое: обычное плавание, короткая петля/вертушка и периодический облет плавно входят и выходят через envelope, чтобы не было скачков позиции в начале/конце анимации.
- Настройки хранятся в `Memorial.sceneJson.soul`:
  - `enabled`;
  - `color`;
  - `glowColor`;
  - `version`.
- Для старых мемориалов без `sceneJson.soul` используется дефолтная включенная душа с цветом `#8ee9ff`.
- На шаге 1 `/create` есть отдельное большое 3D-превью души на фоне неба и выбор цвета.
- Hover/focus по свотчу цвета временно перекрашивает 3D-превью, click сохраняет выбранный цвет в форме; 3D-душа явно пересоздает цветной визуальный слой при смене цвета.
- На шаге 2 `/create` и при редактировании душа снова является частью 3D-сцены и вращается вместе с мемориалом, как на странице мемориала.
- После публикации нового мемориала перед переходом на страницу мемориала запускается `farewell`-анимация: душа облетает мемориал, залетает в домик и растворяется.
- На странице мемориала `/pets/[id]` душа появляется в 3D-сцене и дальше летает рядом с домиком.
- В 3D-режимах `/map` и `/my-pets` душа тоже отображается:
  - у активного мемориала — более заметный/full-вариант;
  - у остальных — облегченный light-вариант, чтобы не перегружать сцену.
- В `/my-pets` 3D координаты души дополнительно переводятся из локальных координат terrain в локальные координаты корневой группы мемориала, иначе души могут визуально собираться в одном месте.
- Важно для будущей оптимизации: при большом количестве мемориалов в одной 3D-сцене лучше оставлять full-анимацию только у активного мемориала.

## 15) Карта `/map`
- Есть режимы `Карта` и `3D`.
- Левый блок фильтров общий для режимов, переключение сглажено CSS transition.
- Карточки списка справа не рендерят 3D; используют служебный 3D-скрин, созданный при публикации мемориала.
- При hover по карточке показывается popup с пользовательской обложкой мемориала, выбранной из фото.
- В карточках показываются только годы жизни.
- В 3D-режиме мемориалы показываются на общей сцене/карусели.
- В 3D-режиме мемориалы отображаются вместе с подарками.
- Hover-подсветка мемориалов в 3D-режиме карты отключена.
- В боковых областях есть hover-стрелки для перелистывания.
- При первом открытии 3D-режима карты камера принудительно выставляется в корректный ракурс.
- Стрелки 3D-карусели карты видны до первого перехода, после первого перехода показываются только при hover по боковым областям.
- В 3D-режиме карты есть loading overlay поверх Canvas до фактической загрузки GLB/текстур.
- Обложка активного мемориала в инфо-блоке 3D-карусели увеличена; в портрете размер задается через `clamp`, чтобы не ломать короткие экраны.
- Индикатор публичности в 3D-инфо-блоке расположен справа сверху рядом с именем, tooltip должен быть поверх соседних панелей.
- При клике на маркер `InfoWindow` не показывается: выбранный маркер анимируется, а информация о мемориале заменяет список мемориалов в панели списка. Кнопка `×` возвращает обычный список.
- На мобильной версии `/map` нельзя допускать скролл всей страницы: скролл должен быть только внутри панелей, а основной map viewport ограничен высотой оставшегося экрана после header.

## 16) Скрин 3D-превью для `/map`
- Делается при публикации мемориала.
- Сохраняется в S3 и используется только для списка мемориалов на `/map`.
- Не добавляется в пользовательскую галерею.
- Делается offscreen без `preserveDrawingBuffer`.
- Разрешение уменьшено примерно в 2 раза относительно первоначального варианта.
- Для скрина используется дополнительный свет и отдельный ракурс камеры.
- После покупки подарка мемориал помечается `needsPreviewRefresh=true`, страница мемориала переснимает preview и загружает его в `/pets/:id/map-preview`.
- При деактивации истекших подарков сервер тоже ставит `needsPreviewRefresh=true`; следующий заход на страницу мемориала переснимет preview уже без истекших подарков.

## 17) `/my-pets`
- Доступ только авторизованным пользователям.
- Оставлены режимы:
  - `4` — список/карточки;
  - `5` — 3D-сцена.

- В карточках:

  - черновики показываются отдельными карточками только при включенном переключателе `Показывать черновики`;

  - fallback `dynamic import` сразу показывает тот же визуальный loading-экран, чтобы при включении 3D не было пустой страницы до загрузки чанка;
  - в портретной мобильной версии loading-окно 3D расположено выше примерно на 15% высоты экрана;
  - при первом открытии 3D-режима автоматически выбирается первый мемориал, чтобы камера сразу вставала в рабочий ракурс;
  - боковые стрелки навигации видны до первого перехода между мемориалами, после этого появляются только при hover по крайним областям;
  - клавиатурные стрелки влево/вправо переключают выбранный мемориал;
  - drag/orbit не должен считаться кликом;
  - инфо-блок выбранного мемориала показывает увеличенное фото, имя, годы одной строкой, индикатор публичности справа в строке имени/дат, эпитафию в кавычках курсивом и кнопку `Открыть мемориал`;
  - в портретном режиме инфо-блок ограничен по высоте и скроллится, чтобы крупная обложка не ломала короткие экраны.
  - на текущей baseline-версии мобильный инфо-блок сделан ниже сверху: высота около `45dvh`, чтобы больше не перекрывать 3D-превью.

## 18) Страница мемориала `/pets/[id]`
- 3D-превью занимает весь экран как фон, интерфейс поверх в стиле game HUD.
- Для страницы мемориала у `MemorialPreview` передается отдельная дефолтная камера/target, чтобы при fullscreen-сцене сам мемориал был выше центра экрана.
- `MemorialPreview` показывает loading overlay с текстом/спиннером/полосой, пока 3D-сцена реально грузится; на `/create` шаге 2 overlay по-прежнему подавляется, потому что там используется отдельный fullscreen loading между шагами.
- Сверху по центру — имя питомца и даты жизни.
- Слева сверху для владельца мемориала есть блок управления:
  - дата активности мемориала;
  - продление по тарифам, включая вариант `Навсегда`;
  - soft-delete мемориала.
- Снизу/сбоку — кнопки для окон:
  - подарки;
  - фотографии;
  - другие мемориалы хозяина;
  - основная информация.
- Блок подарков выдвигается отдельной панелью.
- Список других мемориалов должен скроллиться, если элементов много.
- Header поверх прозрачный.
- Если срок мемориала истек, сервер деактивирует его и он перестает показываться в `/map`, `/my-pets`, списках владельца и лимитах создания.
- В мобильной версии нижние кнопки страницы мемориала не должны уходить под bottom-nav; всплывающие блоки держат безопасный отступ от навигации.
- При открытии блока подарков остальные нижние блоки закрываются, и наоборот.
- Неавторизованный пользователь при попытке почистить грязь не должен попадать на отдельную error-страницу; показывается toast снизу справа с просьбой авторизоваться.
- В мобильном блоке подарков срок подарка и кнопка дарения находятся в одной нижней зоне; при маленькой ширине кнопка должна оставаться внутри рамки без обрезания.
- Вкладки/кнопки информационных блоков на странице мемориала используют светлый фон в стиле общих кнопок и уменьшенную высоту/шрифт.

## 19) Admin `/admin/sql`
- Доступ: `OWNER` и `ADMIN`.
- SQL-консоль поддерживает `SELECT`, `DELETE`, `UPDATE` с ограничениями доступа.

- Есть инструменты:
  - сменить пароль пользователя по email;

## 21) Производительность и стабильность 3D
- Все Canvas сейчас с `dpr={1}`.
- Draco decoder локальный: `/draco/...`.
- Preload моделей в `/create` идет на loading screen перед шагом 2.
- Web обновлен до Next.js `15.5.16` и React `19.2.3`; после такого апгрейда нужно внимательно проверять R3F/Next build, потому что Next может перегенерировать `next-env.d.ts`.
- 3D-режим `/my-pets` подключается через `next/dynamic` только при переключении в режим `3D`; это сильно уменьшает initial JS страницы карточек.
- Для `/my-pets` fallback самого `dynamic import` теперь сразу показывает полноценный loading-экран, чтобы пользователь не видел пустую страницу до загрузки 3D-чанка.
- Помимо перехода на шаг 2, loading overlay есть у:
  - `MemorialPreview` на страницах просмотра/редактирования мемориала;
  - 3D-режима `/my-pets`;
  - 3D-карусели `/map`.
- Важная логика: при hover/выборе детали старая сцена не должна пропадать, пока новая модель не загружена.
- На карте и в списках не нужно массово рендерить 3D-карточки; используется статический 3D-скрин.
- `WebGL context lost` чаще всего связан с большим количеством Canvas/тяжелым preload/памятью GPU.
- CDN пока можно отложить, но при росте трафика его нужно подключить для моделей, иконок, фото и preview.

## 24) Частые проблемы
- Timeweb: env нужны и для build, и для runtime.
- Если Timeweb подставляет unset env как пустые строки, код web теперь трактует пустой `NEXT_PUBLIC_API_URL` как отсутствующий и в production падает обратно на `/api`; `next.config.mjs` так же fallback-ит пустой `INTERNAL_API_URL` на `http://api:3001`.
- Для кириллического IDN в backend/server env лучше использовать punycode `xn--80aeb9a9a9d.com`, потому что часть платформ сериализует origin именно в ASCII-форме.
- Prisma migrations применяются через существующий deploy/build flow; если миграция уже применена, нужен мягкий fallback.
- После изменения Prisma schema локально нужно запускать `pnpm --filter @memorial/api prisma:generate`, иначе API build может не увидеть новые модели.
- Новая миграция wallet/news/documents: `apps/api/prisma/migrations/20260531090000_add_wallet_news_documents/migration.sql`.
- После применения этой миграции появляются таблицы `WalletTransaction`, `NewsPost`, `NewsRead`, `DocumentRevision`.
- R3F JSX типы: в проекте используются касты `const Group = "group" as React.ComponentType<any>` и кастомные refs в `next-env.d.ts`.
- `next build` может перегенерировать `apps/web/next-env.d.ts`; если пропадают R3F refs, нужно вернуть:
  - `/// <reference types="@react-three/fiber" />`
  - `/// <reference path="./types/react-three-fiber.d.ts" />`
- После `next build` также нужно проверить, что в `apps/web/next-env.d.ts` не остались лишние сгенерированные route refs/comment changes, если они не были целью правки.
- 404 на иконках в preload не должен блокировать переход; для отсутствующих иконок нужны заглушки.
- Draco compressed GLB требует локального Draco loader; внешний gstatic не использовать как единственную зависимость.

## 25) Последние активные направления
- Последний зафиксированный рабочий пуш: `be3c36d Add compact font previews and lift gift button`.
- Важный откатный baseline перед июньской мобильной серией: `0765808 Normalize mat house scaling`.
- Текущие правки души:
  - базовая позиция души считается от центра поверхности, сдвинута на `+1` по X, `+1` по Z и находится на уровне `floorY`;
  - `orbit` убран из случайных idle-анимаций;
  - после каждого активного движения ставится спокойный `float` на 13.5-18 секунд;
  - `memorialOrbit` делает вертикальный облет вокруг поверхности, развернут на `-45°`;
  - вход и выход в `memorialOrbit` спиральные, по 4 витка, без дополнительного slow ramp на стыке с основной окружностью;
  - `whirlpool` поднимает душу на `1.5`;
  - прыжки по поверхности и прыжки между двумя точками опущены к уровню `floorY - 0.4`, высота дуги не уменьшалась.
- Детали домиков:
  - детали ставятся в named slots внутри GLB домика (`roof_slot`, `wall_slot`, `sign_slot`, `frame_left_slot`, `frame_right_slot`, `mat_slot`, `bowl_food_slot`, `bowl_water_slot`);
  - общий размер деталей задается при подключении GLB детали к слоту, а итоговый визуальный размер зависит и от самой модели детали, и от slot transform внутри конкретного домика;
  - для `mat_*` домиков масштаб всего домика больше не зависит от длины конкретной текстуры/варианта; все `DOM_mat_1__...` используют множитель как у `DOM_mat_1__base`;
  - для `mat_`-домиков миски еды/воды дополнительно уменьшаются через `getHousePartScaleMultiplier(...)=0.5`, чтобы они выглядели в 2 раза меньше базового размера;
  - для `budka_1` миски ограничены `0.18 x 0.18`, коврик `0.33 x 0.33`.
- В UI выбора домика на создании и редактировании блок домика делится на две видимые зоны: 65% сверху формы домиков, 35% снизу текстуры выбранного домика.
- `toy_` подарки имеют target width `0.5`, `meal_` подарки — `0.45` через `apps/web/lib/gifts.ts`.
- Прозрачные фоны у видимых ghost-кнопок/плашек постепенно заменяются на светлые `bg-white/...` или `#f6efea`, чтобы соответствовать стилю header/menu-кнопок; технические прозрачные hit-area слои 3D-карусели остаются прозрачными.
- Текущие UI-направления после июньской мобильной серии:
  - держать мобильный редактор `/create` шаг 2 и редактирования как две зоны: верхнее 3D-превью и нижний компактный редактор;
