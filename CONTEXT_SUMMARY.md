# Контекст проекта memorial_pitomcev — актуальная сводка

Документ для быстрого входа в проект и продолжения работы без потери контекста.

## 1) Архитектура и инфраструктура
- **Монорепо pnpm workspaces**:
  - `apps/web` — Next.js App Router.
  - `apps/api` — NestJS + Prisma + PostgreSQL.
  - `packages` — общие модули, используется минимально.
- **Деплой**: Timeweb Apps через Docker Compose, сервисы `web` и `api`.
- **Локально**: web `3002`, api `3001`.
- **Хранилище**: Timeweb Object Storage / S3-compatible.
- **3D**: React Three Fiber + drei + three, Draco decoder лежит локально в `apps/web/public/draco`, без зависимости от внешнего CDN.
- **Canvas**: везде выставлен `dpr={1}` для снижения нагрузки на GPU.

## 2) Основные сущности и фичи
- Пользователи, питомцы, мемориалы, фотографии, подарки, размещения подарков, wallet.
- Мемориал может быть публичным или приватным.
- Публичные мемориалы отображаются на карте.
- Есть внутренняя валюта, пополнение баланса и покупка подарков.
- 20% от пополнения баланса начисляется в благотворительный фонд проекта.
- Есть страница благотворительности с отчетами и админским созданием отчетов с фото через S3.
- Есть загрязнение мемориала: грязь накапливается по времени с момента создания/последней очистки, хранение состояния на сервере.
- По умолчанию пользователь может создать максимум 5 мемориалов; для owner действует лимит 10000. Лимит можно менять в `/admin/sql` по email.
- Мемориалы имеют срок активности (`Memorial.activeUntil`) и могут быть деактивированы без физического удаления данных.
- Удаление мемориала пользователем — soft-delete: мемориал скрывается из списков/карты/лимитов, но записи остаются в БД.

## 3) Доступы и роли
- Есть уровни доступа:
  - `OWNER` — владелец проекта: `andreyvbvbvb@gmail.com` / login `andreyvbvbvb`.
  - `ADMIN` — админ через поле `role=ADMIN` в БД.
  - `USER` — обычный пользователь.
- `OWNER` может назначать и снимать админов через `/admin/sql` по email.
- `ADMIN` имеет доступ к админским инструментам, но не может:
  - понижать owner;
  - назначать новых admin;
  - менять owner через SQL/смену пароля.
- Временные калибровочные инструменты в `/create` показываются только `OWNER` и `ADMIN`.

## 4) Auth, документы и профиль
- При регистрации пользователь должен принять пользовательское соглашение и публичную оферту.
- Для уже существующих пользователей после авторизации показывается обязательное окно принятия документов, если флаги не проставлены.
- Документы-заглушки размещены на `/about`:
  - `#agreement`
  - `#offer`
- На `/auth` нет кнопки входа в header.
- На `/profile` убраны кнопки `Мои питомцы` и `Создать мемориал`.
- Восстановление пароля: показывается инструкция написать на почту поддержки.

## 5) Общий UI и Header
- Название сервиса в header: **МяуГав**.
- Header прозрачный/полупрозрачный, лежит поверх страниц.
- Кнопка создания: `+ создать`, темная CTA, плюс анимируется при hover; перед переходом проверяет лимит мемориалов пользователя.
- Кнопки `Мои питомцы`, `Карта`, меню-книжка приведены к общему компактному стилю.
- Меню-книжка содержит:
  - профиль/аккаунт;

  - плюс пополнения расположен рядом с суммой баланса и анимируется как плюс в `+ создать`;
  - ссылки на `/charity`, `/about`, `/admin/sql` при доступе;
  - logout.
- Текущий шрифтовой стек:
  - `Avenir Next`
  - `Segoe UI`
  - `Helvetica Neue`
  - `system-ui`
- Планировалось подключить локальный `Nunito` после добавления файлов шрифта.

## 6) Создание мемориала `/create`
- Мастер сейчас состоит из **2 шагов**.
- **Шаг 1**:
  - центрированный декоративный блок основных данных;
  - поля: имя, вид питомца, дата рождения, дата ухода;
  - кнопка `Продолжить` со стрелкой-анимацией;
  - фон должен быть единым с header, без лишнего скролла.
- Между шагами — непрозрачный fullscreen loading screen:
  - progress bar;
  - подсказки сервиса;
  - подсказки управляются через админку (`LoadingTip`).
- **Шаг 2**:
  - 3D-превью на весь экран как фон;
  - справа — редактор деталей мемориала;
  - слева снизу — кнопки окон: основные данные, история/эпитафия, карта/маркер, фото;
  - окна от левых кнопок располагаются справа от панели кнопок и не должны накладываться на сами кнопки;
  - справа снизу — кнопка `Завершить`, открывает модал проверки и оплаты;
  - баланс подгружается заново при каждом открытии проверки.
- Окна шага 2 закрываются кликом вне окна.
- Кнопки слева подсвечиваются, пока пользователь хотя бы раз не открывал соответствующий блок; вместе с подсветкой показывается `!`.
- Временные калибровочные слайдеры позиции/масштаба домика видны только admin/owner.
- Для публикации требуется минимум одна фотография питомца.

## 7) Данные в `/create`
- **Основные данные** можно открыть и поправить на шаге 2.
- **История/эпитафия** — отдельное окно.
- **Маркер**:
  - карта слева;
  - справа категории маркеров по видам питомцев;
  - категории — вертикальные иконки с tooltip;
  - можно кликнуть по карте или вручную ввести координаты;
  - есть чекбокс публичности с tooltip.
- **Фото**:
  - максимум 10 фото;
  - до 6 МБ каждое;
  - пользователь выбирает фото-обложку для popup маркера;
  - служебный 3D-скрин мемориала не попадает в галерею и используется отдельно для списка на `/map`.
  - скрин делается от фиксированной базовой позиции камеры, а не от текущего пользовательского ракурса.
- **Поверхность / сезоны**:
  - seasons доступны как варианты terrain;
  - переключатель авто-смены сезонов по умолчанию выключен;
  - сезоны представлены цветными квадратиками.
- **Превью подарков**:
  - чекбокс `Посмотреть с подарками` показывает пример заполнения слотами подарков.

## 8) 3D: terrain, house, parts
- Terrain и house — отдельные GLB.
- В terrain обязателен `dom_slot`.
- Домик ставится в `dom_slot`, затем к нему применяются:
  - fit-scale;
  - индивидуальная позиция/поворот/scale из `apps/web/lib/house-layout.ts`;
  - при необходимости override по terrain.
- Текущие группы домиков: `budka_*`, `kotik_*`, `mat_*`.
- Названия в UI:
  - `budka_N` → `Будка N`;
  - `kotik_N` → `Котик N`;
  - `mat_N` → `Матрас N`.
- Поддерживаются house textures через отдельные GLB вида `DOM_{baseId}__{texture}.glb`.
- Для базовой текстуры используется `DOM_{baseId}__base.glb` и она тоже показывается в выборе текстуры.
- Иконки домиков и текстур — заглушки/картинки в `apps/web/public/memorial/options/house` и `house-texture`.

## 9) Слоты деталей домика
Слоты берутся автоматически из GLB домика по именам узлов:
- `roof_slot`
- `wall_slot`
- `sign_slot`
- `frame_left_slot`
- `frame_right_slot`
- `mat_slot`
- `bowl_food_slot`
- `bowl_water_slot`

Если слот отсутствует в GLB, соответствующая категория выбора в редакторе не показывается.

## 10) Подарки
- Слоты подарков в моделях:
  - `gift_{type}_{index}` — основной формат;
  - legacy `gift_slot_1..` читается как `ground`.
- Дефолтный список слотов — 10, но рендерятся только реальные anchors в GLB.
- Заглушка слота: `apps/web/public/models/gifts/slot_placeholder.glb`.
- Типы подарков включают `toy`, `meal`, `birds`, `flower`, `candle`, `star` и др. по структуре папок.
- Иконки подарков: `apps/web/public/gifts_icons/{code}.png`.
- В списке подарков карточка: картинка занимает карточку, цена в кружке снизу по центру.
- Срок подарка выбирается ползунком.
- Истекшие подарки не удаляются из БД, а деактивируются через `GiftPlacement.isActive=false`, `deactivatedAt`, `deactivationReason=expired`.
- На странице мемориала есть чекбокс подсветки доступных мест под подарки; при выборе конкретного подарка подсвечиваются подходящие слоты независимо от чекбокса.
- Автогенерация подарков обновляет web/api generated-файлы при `prebuild`.

## 11) Загрязнение мемориала
- На странице мемориала грязь считается сервером от даты создания или последней очистки.
- Каждые 2 недели добавляется следующий уровень, максимум 4.
- Очистка кликом по грязи или кнопкой чистит все уровни сразу и обновляет дату очистки.
- Сейчас используются отдельные модели по слотам:
  - базовый fallback: `apps/web/public/models/dirt/dirt_1.glb` ... `dirt_4.glb`;
  - кастомно для домика: `apps/web/public/models/dirt/{baseId}/dirt_1.glb` ... `dirt_4.glb`.
- В самих моделях также поддерживается логика сегментов `dirt_1..dirt_4`, если используется единая GLB-модель с сегментами.

## 12) Автогенерация моделей и данных
- Скрипт: `apps/web/scripts/generate-models.mjs`.
- Запускается в `predev` и `prebuild`.
- Генерирует:
  - `apps/web/lib/memorial-models.generated.ts`
  - `apps/web/lib/memorial-options.generated.ts`
  - `apps/web/lib/memorial-slots.generated.ts`
  - `apps/web/lib/markers.generated.ts`
  - `apps/web/lib/gifts.generated.ts`
  - `apps/api/src/gifts/gifts.generated.ts`
- Поэтому после добавления `.glb` или картинок обычно достаточно запустить dev/build или вручную `node apps/web/scripts/generate-models.mjs`.

## 13) Папки моделей и иконок
- Terrains: `apps/web/public/models/terrains`
  - формат: `TERRAIN_{id}_{season}.glb`, например `TERRAIN_2_winter.glb`.
- Houses: `apps/web/public/models/houses`
  - формат: `DOM_{baseId}__{texture}.glb`, например `DOM_budka_1__base.glb`.
- Parts: `apps/web/public/models/parts/*`
  - `roof`, `wall`, `sign`, `frame-left`, `frame-right`, `mat`, `bowl-food`, `bowl-water` и т.д.
- Gifts: `apps/web/public/models/gifts/{type}`.
- Dirt: `apps/web/public/models/dirt/{baseId}/dirt_1..4.glb` + общий fallback `models/dirt/dirt_1..4.glb`.
- Превью-иконки деталей `/create`:
  - URL: `/memorial/options/{category}/{id}.png`.
  - путь: `apps/web/public/memorial/options/{category}/{id}.png`.
  - пример: `bowl_food_3` → `memorial/options/bowl-food/bowl_food_3.png`.
- Иконки поверхностей:
  - `apps/web/public/memorial/options/environment/{id}.png`.
- Иконки домиков:
  - `apps/web/public/memorial/options/house/{baseId}.png`.
- Иконки текстур домиков:
  - `apps/web/public/memorial/options/house-texture/{variantId}.png`.

## 14) Карта `/map`
- Есть режимы `Карта` и `3D`.
- Левый блок фильтров общий для режимов, переключение сглажено CSS transition.
- Карточки списка справа не рендерят 3D; используют служебный 3D-скрин, созданный при публикации мемориала.
- При hover по карточке показывается popup с пользовательской обложкой мемориала, выбранной из фото.
- В карточках показываются только годы жизни.
- В 3D-режиме мемориалы показываются на общей сцене/карусели.
- В 3D-режиме мемориалы отображаются вместе с подарками.
- Hover-подсветка мемориалов в 3D-режиме карты отключена.
- В боковых областях есть hover-стрелки для перелистывания.
- При первом открытии 3D-режима карты камера принудительно выставляется в корректный ракурс.
- Стрелки 3D-карусели карты видны до первого перехода, после первого перехода показываются только при hover по боковым областям.

## 15) Скрин 3D-превью для `/map`
- Делается при публикации мемориала.
- Сохраняется в S3 и используется только для списка мемориалов на `/map`.
- Не добавляется в пользовательскую галерею.
- Делается offscreen без `preserveDrawingBuffer`.
- Разрешение уменьшено примерно в 2 раза относительно первоначального варианта.
- Для скрина используется дополнительный свет и отдельный ракурс камеры.
- После покупки подарка мемориал помечается `needsPreviewRefresh=true`, страница мемориала переснимает preview и загружает его в `/pets/:id/map-preview`.
- При деактивации истекших подарков сервер тоже ставит `needsPreviewRefresh=true`; следующий заход на страницу мемориала переснимет preview уже без истекших подарков.

## 16) `/my-pets`
- Доступ только авторизованным пользователям.
- Оставлены режимы:
  - `4` — карточки;
  - `5` — 3D-сцена.
- Переключатель `Карточки / 3D` сейчас расположен снизу слева.
- В карточках:
  - стиль по мотивам `visual_code_3`;
  - фото/preview сверху;
  - бейдж публичности поверх фото;
  - даты показываются годами.
- В 3D-режиме:
  - сцена fullscreen;
  - фон неба подключен;
  - увеличены расстояния между мемориалами;
  - при первом открытии 3D-режима автоматически выбирается первый мемориал, чтобы камера сразу вставала в рабочий ракурс;
  - боковые стрелки навигации видны до первого перехода между мемориалами, после этого появляются только при hover по крайним областям;
  - drag/orbit не должен считаться кликом;
  - инфо-блок выбранного мемориала показывает крупное фото, бейдж, имя, годы, эпитафию в кавычках курсивом и кнопку `Открыть мемориал`.

## 17) Страница мемориала `/pets/[id]`
- 3D-превью занимает весь экран как фон, интерфейс поверх в стиле game HUD.
- Сверху по центру — имя питомца и даты жизни.
- Слева сверху для владельца мемориала есть блок управления:
  - дата активности мемориала;
  - продление на 1 год;
  - soft-delete мемориала.
- Снизу/сбоку — кнопки для окон:
  - подарки;
  - фотографии;
  - другие мемориалы хозяина;
  - основная информация.
- Блок подарков выдвигается отдельной панелью.
- Список других мемориалов должен скроллиться, если элементов много.
- Header поверх прозрачный.
- Если срок мемориала истек, сервер деактивирует его и он перестает показываться в `/map`, `/my-pets`, списках владельца и лимитах создания.

## 18) Admin `/admin/sql`
- Доступ: `OWNER` и `ADMIN`.
- SQL-консоль поддерживает `SELECT`, `DELETE`, `UPDATE` с ограничениями доступа.
- Есть быстрые запросы, просмотр таблиц/колонок.
- Есть инструменты:
  - сменить пароль пользователя по email;
  - выставить лимит мемориалов по одному или нескольким email;
  - удалить/отключить питомца, фото и мемориал по id;
  - CRUD подсказок загрузки (`LoadingTip`);
  - управление admin-доступом доступно только owner.

## 19) Благотворительность `/charity`
- 20% от пополнения баланса учитывается как сумма, которая должна пойти на благотворительность.
- На публичной странице показываются только:
  - сколько уже направлено на благотворительность;
  - отчеты.
- Блоки `сколько начислено/накопили` и `осталось отдать` скрыты из публичного UI.
- Admin/owner может добавлять отчет:
  - заголовок;
  - сумма;
  - текст;
  - фотографии;
  - дата создания ставится автоматически.
- Фото отчетов грузятся в S3.
- Фото каждого отчета открываются в полноэкранном просмотрщике с навигацией назад/вперед, как фото на странице мемориала.
- Ссылка на `/charity` есть в меню-книжке header.

## 20) Производительность и стабильность 3D
- Все Canvas сейчас с `dpr={1}`.
- Draco decoder локальный: `/draco/...`.
- Preload моделей в `/create` идет на loading screen перед шагом 2.
- Важная логика: при hover/выборе детали старая сцена не должна пропадать, пока новая модель не загружена.
- На карте и в списках не нужно массово рендерить 3D-карточки; используется статический 3D-скрин.
- `WebGL context lost` чаще всего связан с большим количеством Canvas/тяжелым preload/памятью GPU.
- CDN пока можно отложить, но при росте трафика его нужно подключить для моделей, иконок, фото и preview.

## 22) Частые проблемы
- Timeweb: env нужны и для build, и для runtime.
- Prisma migrations применяются через существующий deploy/build flow; если миграция уже применена, нужен мягкий fallback.
- R3F JSX типы: в проекте используются касты `const Group = "group" as React.ComponentType<any>` и кастомные refs в `next-env.d.ts`.
- `next build` может перегенерировать `apps/web/next-env.d.ts`; если пропадают R3F refs, нужно вернуть:
  - `/// <reference types="@react-three/fiber" />`
  - `/// <reference path="./types/react-three-fiber.d.ts" />`
- 404 на иконках в preload не должен блокировать переход; для отсутствующих иконок нужны заглушки.
- Draco compressed GLB требует локального Draco loader; внешний gstatic не использовать как единственную зависимость.
